## 2026-07-23: doctrine: an authored skill stands alone — no cross-invocation, no assumed installs

- **Owner directive, 2026-07-23:** skills the kitchen produces kept referencing other skills and, worse, directing the agent to invoke them or assuming they were installed. That is a no-go. An authored skill stands alone: it never instructs invoking another skill and never assumes one is present — not a third-party skill, and not a harness built-in (`/loop`, `/batch`, `/code-review`) either. The behavior another skill happens to package is encoded inline ("repeat until nothing is pending", not "drive it with `/loop`"), so the guidance holds in any harness with whatever skills are installed, or none. The only surviving cross-skill reference is an optional, one-directional "see also" pointer that costs nothing when the target is absent.
- **Stance reversal, mechanics kept.** This overturns the `references/skills.md` "Skills invoking skills; harness-capability clauses" section (added 2026-07-15), which taught the opposite — that a body "may direct invoking another skill when that's the honest mechanism," packaged as a portable "capability clause" with `/loop` as its worked example. The owner chose *absolute, no exceptions* over a narrow built-in carve-out (`AskUserQuestion`, this session). The section's underlying facts were true and are retained, reframed from a licence into the reasons the dependency is fragile: skill invocation is prompt injection that inflates standing context; model-invocability is per-skill config, not a guarantee (a target with `disable-model-invocation: true` fails the chain silently, bundled built-ins go off wholesale via `disableBundledSkills`, `user-invocable: false` changes only the menu not Skill-tool access, and model-invocable listings vary by environment). Section retitled *A skill stands alone*; the redundant compaction-re-attach restatement it carried was dropped (already stated in Lifecycle).
- **Where it landed.** forge `SKILL.md` — the "Blast radius of one skill" bullet generalized to "A skill stands alone" so the rule is salient in the body where skills are actually designed (the ingest-back-reference case it used to cover is folded in). forge `references/skills.md` — the section rewrite above, plus a new *Cross-skill dependency* anti-pattern ("first run `/X`" breaks the moment `/X` is absent, disabled, or off-menu). ingest `SKILL.md` — the "skill used as a source … never cross-wired as a dependency" line extended so the artifact forge builds from a skill-source stands alone. harness-audit `SKILL.md` — Self-consistency (check 1) gains the dead cross-reference find: a skill/command invoking or assuming an uninstalled `/X` reads as working capability but fails silently; scoped to the reference's *presence*, not body quality, which stays the forge's remit.
- **The kitchen's own process family is untouched and intended.** forge ↔ ingest ↔ harvest still point at and hand off to one another; per the owner these mentions are reference material, and as co-shipped pipeline skills their pointers are the permitted one-directional "see also" form, not assumed third-party installs.

## 2026-07-21: forge: four AGENTS.md-routing patterns for doc-fleet repos

- **forge `references/always-on.md` gains doc-fleet routing discipline**, folded into the existing "When AGENTS.md is primary" paragraph and its Escalation bullet rather than a new section.
