agent-kitchen

This is a side project, about a year old. It started with corrections: when I caught myself telling Claude Code the same thing a second time, I wrote it down where the agent would see it. Many of the skills here started that way, as a sentence I was tired of saying twice. The repo part came later, when I started asking which of those sentences were still needed, and how I'd know.

Like everyone, I was reading a lot about how to work with these things. At some point I noticed where all that advice ends up: the model's default is the consensus of it, and whatever a post can teach everyone comes free with a later release. So I let the feed go. The facts I still need, like what changed in Claude Code and when, the harness verifies against each release and keeps as a snapshot. Keeping up became the harness's job.

I kept the trending vocabulary out too. A borrowed name carries the most popular version of the idea it names, and if I'd built under those names I'd have drifted into building the popular things. So I described what I wanted in the words I already had. It's why things in this repo get cooked and tasted and go stale, instead of getting orchestrated.

Most of the year went into conversations with nothing at the end of them. I asked the model what it actually does with an instruction, why a skill changed nothing, what it would delete from this repo if I let it. I brought it writing that had nothing to do with AI. I challenged its answers and let it challenge mine, and whatever neither of us could break got written down.

My own taste went in on the same terms: each opinion I write down becomes a test the bare model takes on every release, and the ones it catches up to get deleted. I never had to take my own word for it.

The model gets better every few months without any help from me. But what I want, it can only get from me. This repo is a year of telling it.

What it is

Claude Code reads a layer of files before it touches your code: skills, hooks, rules, CLAUDE.md, workflow scripts. That layer decides a surprising amount of what the agent actually does, and most people write it once and never look at it again. agent-kitchen treats that layer as the product. Most repos configure a project; this one configures the thing that configures it, which is why I call it a second-order harness. It runs as two plugins under one marketplace. COOK makes the tools, tastes them, dates them, throws out what's gone stale. SERVE is what has survived so far. It still tastes better than store-bought.

The premise

Ask a frontier model for anything and you get the most probable competent version of the thing: the median commit message, the median review, the median skill. That's what a model is. So a harness that works is a short list of places where what you want is not what's most probable, and every artifact here is one entry on that list: a commitment that drags the model off its median in a direction I chose.

Each one competes with two forces: the frontier model, which absorbs generic craft with every release, and my time, which every artifact charges rent on. Any rule a stranger could have written is on borrowed time.

One artifact, start to finish

One of the skills in SERVE is called speed-daemon. It exists because the React a bare model writes waits for the network: click, spinner, then the interface moves. The skill flips that default. The interface moves first, the network catches up, and a failed request rolls the change back with a visible error. Underneath it is one opinion: an app should feel like Linear, and this is the component-layer pattern that gets you there.

It earned its place on 2026-05-22, when Opus 4.7 kept writing the waiting version. The skill carries that fact as a one-line pin, and it keeps the failures that earned it as five runnable probes: the verbatim prompt, what the bare model got wrong, what passing looks like. Everything else, why it exists and every verdict since, lives in CHANGELOG.md. A fix only means something relative to the model it fixed.

The probes are what keep the claim honest, and they've already paid for themselves twice. A week after the skill shipped, Opus 4.8 passed the easiest probe unaided, and a one-probe check would have deleted a skill that was still needed; the other four probes still failed, so the rule became that nothing dies until every probe passes. Then the first Fable 5 replay found that same easy case failing again. Absorption runs backwards too, so probes never retire while the skill lives. The verdict this release: four of five still earning, KEPT.

The day all five pass unaided, the model has absorbed the job and the skill gets deleted. That has already happened once elsewhere in the kitchen: a workflow-design skill died when the tool it wrapped began carrying the same guidance in its own description.

Cook

COOK is where artifacts come from, and it runs as a loop. It mines session transcripts for the corrections I had to make twice. Once is conversation; twice is a failure worth keeping, and the no-skill baseline already ran, in production, in those sessions. The forge routes each failure to the surface that should hold it (a skill, a hook, a path-scoped rule, a line of CLAUDE.md) and holds everything to one bar: earned by a real failure or a real source, shifts the agent's attention in a way you can name, beats the model's default. Probes keep the earning failure runnable. And because everything the agent loads at session start is context some task can't use, an audit prices what each artifact costs to carry and removes what doesn't cover its rent.

The forge's hardest question is whether an artifact is additive or transformative: does it save me typing, or does it change what the agent can think of doing? Only the second kind earns its context.

Serve

SERVE carries the standing claim that everything in it beats the unaided current model at its job, provably, the way speed-daemon just did. On each model release every probe in the plugin replays against the bare model in a clean session, and the verdicts land in the changelog: keep, revise, delete.

Provably is only as strong as the pass call, so the pass criterion gets written the day the failure happens, as things a judge can check in the output, never adjectives. Skills that encode taste get two verdicts per replay, the mechanical one and mine, and mine wins. When the two disagree, that's a finding: an output that passes every written tell and still reads wrong means the tells are missing one, and the gap gets harvested. What taste means here is concrete. One probe asks for animation on a surface people hit a hundred times a day; the bare model delivers it, with real craft and zero pushback. Knowing how is absorbed. Knowing when not to is still mine.

How it was made

All of it was made in conversation with the thing it configures. The artifacts are the residue of the arguments: each one points to a session where the bare model failed and the fix proved worth keeping. The artifact itself never mentions the chat that produced it, though; the next reader was never there.

CN Would this rule change anything you'd actually do?
✳ Not much. The default already covers it.
CN Then it doesn't ship. What would change something?
✳ Tell me what you'd delete from your own writing. That I can't guess.
most artifacts die here, before they're written

There's no standard for any of this yet; nobody has kept one coding agent configured across enough model releases for one to form. What I have is the idea from the top: the model hands you the most probable version of whatever you ask for, and the harness is the short list of places where I want something else. Neither side can write that list alone. I can't see the distribution from where I stand; the model can't guess my deviations from where it does. So the list gets written in conversation, and the pins, probes and verdicts are the bookkeeping that keeps it true while the base underneath it moves.

The next model release will delete some of these artifacts. What's left is what the model still can't have.
